🔧 Pre-Demo Setup Checklist

Complete these before the audience joins. Aim for 15 minutes prep time.

🖥️ Environment

  • ADO project open: https://dev.azure.com/brandsafway1/Project1
  • GitHub repo open: github.com/sautalwar/brandsafway-copilot-demo
  • VS Code open with repo loaded + Copilot Chat visible
  • Terminal open in repo root — run npm test to confirm 6/6 pass
  • Browser tabs arranged: ADO Boards → GitHub Repo → GitHub Actions → ADO Pipelines

✅ Configuration

  • GitHub Copilot coding agent enabled for the repo
  • ADO ↔ GitHub connection active (Project Settings → GitHub connections)
  • Branch protection rules on main (require PR, 1 reviewer, CI checks)
  • ADO Pipeline connected to repo (uses azure-pipelines.yml)
  • GHAS features enabled: Dependabot, CodeQL, Secret scanning
  • Labels exist on repo: copilot-agent, ado-synced, severity:high

🎯 Demo Task to Prepare

The demo will use this enhancement request:
"Add a GET /api/inspections/stats endpoint that returns summary statistics — total count, count by status, count by severity, and recent inspection count."

This TODO already exists in src/app.js. The Copilot agent will implement it, write tests, and raise a PR — all autonomously.

1 Create ADO Work Item

3 min

🟢 Do

  • Open ADO Boards
  • Click + New Work Item → User Story
  • Title: Add inspection statistics endpoint to Safety Tracker API
  • Description: "As a safety manager, I need a summary statistics endpoint so I can see total inspections, breakdowns by status and severity, and recent activity at a glance."
  • Set: Priority = 2, Severity = Medium, Iteration = current sprint
  • Note the Work Item ID (e.g., 42) — you'll need this

🔵 Say

"This is where every request starts — right here in Azure DevOps, where your teams already plan and track work. Nothing changes about your planning process. The difference is what happens next."

✨ Aha Moment: "Your existing ADO workflow is the starting point — we're not replacing it, we're accelerating everything downstream."

2 Create GitHub Issue + ADO Link

3 min

🟢 Do

  • Open GitHub → New Issue
  • Title: Add GET /api/inspections/stats endpoint AB#42
    (replace 42 with your actual work item ID)
  • Body: 
    ## Description
Add a `GET /api/inspections/stats` endpoint that returns summary statistics.

## Acceptance Criteria
- Returns `total` count of all inspections
- Returns `byStatus` object with counts per status (open, in-review, closed, scheduled)
- Returns `bySeverity` object with counts per severity (low, medium, high)
- Returns `recentCount` of inspections from last 30 days
- All new code covered by Jest + Supertest tests

## Azure DevOps Reference
AB#42 — https://dev.azure.com/brandsafway1/Project1/_workitems/edit/42
  • Add label: copilot-agent
  • Click Submit new issue

🔵 Say

"Notice the AB#42 in the title — that's the magic link. Azure Boards automatically picks this up and connects the GitHub issue to the ADO work item. Open your ADO work item and you'll see it in the Development section. Two systems, one source of truth."

🟡 Expect

  • GitHub issue created with copilot-agent label
  • ADO work item's Development panel shows the linked GitHub issue
  • The auto-assign-copilot.yml workflow fires automatically

💡 Aha Moment

"Zero manual linking" — the AB# pattern gives you full traceability between ADO and GitHub with no copy-paste, no context switching. Your compliance and audit trail is built-in.

3 Copilot Agent Gets Assigned

2 min

🟢 Do

  • Show the issue page — point out the copilot-agent label
  • Click Actions tab → show the auto-assign-copilot workflow running
  • Return to the issue — Copilot should now be listed as assignee
  • Show the automated comment: "🤖 GitHub Copilot coding agent has been assigned…"

🔵 Say

"The moment we added the copilot-agent label, a GitHub Actions workflow automatically assigned Copilot as the developer. This is the trigger — the agent is now reading the issue, understanding the codebase, and planning its implementation. No human developer needed to pick this up."

✨ Aha Moment: "A label is all it takes. Your team decides which work goes to the agent — Copilot handles the how. Human judgment drives what gets automated."

4 Agent Creates a Branch

Part of 8 min block

🟢 Do

  • On the issue page, look for the Copilot agent's session link / activity indicator
  • Click into the Copilot coding agent session to show live progress
  • Point out: the agent created a branch (e.g., copilot/7-add-stats-endpoint)

🔵 Say

"The agent follows our branch naming convention from copilot-instructions.md — same standards your developers follow. It branched from main, just like any team member would."

💬 Fill Time — Show Copilot Instructions

While the agent works, open .github/copilot-instructions.md in VS Code and walk through it:
"This is our guardrails file. It tells the agent our coding standards, testing requirements, how to name branches, how to format commits with AB# references, and security rules. Think of it as your team's coding playbook — but for AI."

5 Agent Develops the Enhancement

Part of 8 min block

🟢 Do

  • Show the agent's live coding session (if visible in the Copilot UI)
  • Narrate what the agent is doing: reading the existing code, understanding the data model, implementing the stats endpoint

🔵 Say

"Watch what's happening — the agent read the issue, then explored src/app.js and src/data/inspections.js to understand the existing patterns. Now it's writing the stats endpoint following the same Express patterns already in the codebase. It's not just generating code — it's understanding context."

💬 Fill Time — VS Code Copilot Demo

While waiting, switch to VS Code and demonstrate:

  • Copilot Chat: Ask "Explain the inspection data model" → shows Copilot understands the codebase
  • Copilot CLI: In terminal, show gh copilot suggest "how to add a new route in Express"
  • Inline Completions: Start typing a new function in app.js — show ghost text suggestions

"Three surfaces — Chat, CLI, and inline — same AI, same context. Your developers get help wherever they work."

6 Agent Commits Code + Tests

Part of 8 min block

🟢 Do

  • Show the agent's commit(s) on the branch — click the branch name on GitHub
  • Point out: commit messages include AB#42 reference
  • Show the test file changes — the agent wrote tests, not just production code
  • Highlight: the agent ran npm test and confirmed all tests pass before proceeding

💡 Aha Moment

"Tests are not optional" — the copilot-instructions.md requires tests for every change, and the agent follows that rule. Same quality bar as your human developers, enforced automatically.

✨ Aha Moment: "The commit message says AB#42. Go back to your ADO work item — you'll see this commit linked in the Development panel. Full traceability, zero effort."

7 Agent Raises a Pull Request

3 min

🟢 Do

  • Open the PR the agent created — navigate to Pull Requests tab
  • Walk through the PR description:
    • Summary of changes
    • Closes #<issue> reference
    • AB#42 ADO reference
    • Testing section describing what was tested
  • Show that CI checks are already running (GitHub Actions)
  • Show that the ADO pipeline was also triggered

🔵 Say

"The agent didn't just dump code into a PR — it wrote a proper description, linked the issue, referenced the ADO work item, and described its testing. This PR looks exactly like what you'd expect from a senior developer. And it triggered both your GitHub CI and your Azure DevOps pipeline automatically."

✨ Aha Moment: "Look at this PR. If I hadn't told you, could you tell an AI wrote it? That's the point — same process, same quality, same traceability. Just faster."

8 Human Reviews + Approves

5 min

🟢 Do

  • Click Files changed on the PR
  • Walk through the code changes — show the stats endpoint implementation
  • Show the test coverage — verify it tests each statistic field
  • Optionally leave a review comment on a specific line to show the feedback flow
  • Click Review changes → Approve

🔵 Say

"This is the human-in-the-loop moment. The agent wrote the code, but a human — your dev lead — reviews and approves. Branch protection rules enforce this: no PR merges without at least one human approval and all CI checks passing. The AI accelerates; humans govern."

💡 Aha Moment

"Human-in-the-loop by design" — Copilot can't bypass your review process. Branch protection is enforced for everyone, including AI agents. Your governance stays intact.

💬 Talking Point

"Think about what just happened: your developer's job shifted from writing code to reviewing code. They're now focused on quality, architecture, and business logic — the high-value work."

9 ADO Pipeline Validates

4 min

🟢 Do

  • Open ADO Pipelines
  • Show the pipeline run triggered by the PR
  • Walk through the stages:
    • Build & Test — npm ci, npm test (all tests pass), lint
    • Security Scan — npm audit for dependency vulnerabilities
    • Deploy — will run on merge to main
  • Switch back to GitHub — show the ADO pipeline check status on the PR

🔵 Say

"Your existing Azure DevOps pipeline is part of the quality gate. GitHub CI runs the tests, ADO pipeline validates independently — two checkpoints, both must pass. The agent's code gets the same scrutiny as any developer's code. No shortcuts."

✨ Aha Moment: "Same pipeline, same rules, same checks. The agent doesn't get a free pass — it earns its merge the same way your developers do."

10 Merge + Deploy

3 min

🟢 Do

  • On the PR page, all checks show green ✅
  • Click Merge pull request → Confirm merge
  • Show the issue auto-closes (because of Closes #<issue>)
  • Open the ADO work item → show the PR and merge commit linked in Development
  • Show the Deploy stage kicks off in ADO Pipelines

🔵 Say

"And that's the full loop. ADO work item → GitHub issue → Copilot agent → branch → code → tests → PR → human review → pipeline validation → merge → deployment. Every step traceable, every step governed, every step auditable. What used to take a developer a day just happened in minutes."

🎯 Key Aha Moment: "Look at the ADO work item now. It shows the linked issue, the branch, every commit, the PR, the pipeline result, and the deployment. Full end-to-end traceability — from backlog to production — and the developer didn't write a single line of code manually."

📊 The Big Picture

Open ADO and GitHub side by side. Show the complete audit trail:
ADO Work Item → linked GitHub Issue → linked Branch → linked Commits → linked PR → Pipeline Run → Deployment

"This isn't just speed. This is compliance. This is auditability. This is how regulated industries like yours should be adopting AI."

⭐ Bonus: GitHub Advanced Security

6 min

🟢 Do — Dependabot

  • Open repo Security tab → Dependabot alerts
  • Show any existing alerts for npm dependencies
  • Open .github/dependabot.yml — show the automated PR schedule
  • "Dependabot doesn't just alert — it opens PRs with the fix."

🟢 Do — CodeQL

  • Open Security tab → Code scanning
  • Show the CodeQL workflow in Actions tab
  • If any findings exist, walk through one
  • "Every PR — including Copilot's — gets scanned for vulnerabilities before it can merge."

🟢 Do — Secret Scanning

  • Open repo Settings → Code security → Secret scanning
  • Show it's enabled — GitHub scans every push for leaked credentials
  • "If anyone — human or AI — accidentally commits an API key, it gets caught instantly."

💡 Aha Moment

"Shift-left security, automated" — Three layers of security scanning run on every change: dependency vulnerabilities (Dependabot), code vulnerabilities (CodeQL), and credential leaks (Secret Scanning). The Copilot agent's code gets the same security treatment as any developer's.

🛡️ Aha Moment: "You're not choosing between speed and security. GitHub Advanced Security means AI-written code gets more scrutiny, not less. Security scales with velocity."
