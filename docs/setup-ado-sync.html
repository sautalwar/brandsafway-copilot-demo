Automation Path

Azure DevOps work item to GitHub issue to Copilot agent

This guide covers the full BrandSafway demo automation path: create or update a work item in Azure DevOps, send the event into GitHub Actions, create a synchronized GitHub issue with an AB# reference, and assign the Copilot coding agent so work can begin with minimal manual handoff.

Source system

Azure DevOps project: https://dev.azure.com/brandsafway1/Project1

Automation target

GitHub repo: github.com/sautalwar/brandsafway-copilot-demo

Workflow entry point

GitHub Actions workflow: ado-issue-sync.yml with repository_dispatch and workflow_dispatch support.

1. Overview

This automation is designed for a full ADO → GitHub execution path where Azure Boards remains the planning system and GitHub becomes the execution system for Copilot-enabled development.

What the automation does

  1. An Azure DevOps work item is created or updated.
  2. An Azure DevOps service hook posts a repository_dispatch event to GitHub.
  3. The GitHub Actions workflow ado-issue-sync.yml runs.
  4. A GitHub issue is created with the related AB#<work item id> reference in the title or body.
  5. The issue is labeled for sync and severity, then the Copilot coding agent is assigned.

Use this full automation path when

  • You want Azure DevOps work item creation to start the GitHub execution flow automatically.
  • You need a repeatable demo or operating model with minimal manual steps.
  • You want the ADO work item and GitHub issue linked for traceability.

Use the manual path instead when

  • You do not have service hooks or PAT-based integration approved yet.
  • You only need to assign Copilot on issues already created in GitHub.
  • You want a lighter setup using the companion manual guide setup-auto-assign.html.

2. Prerequisites

Confirm the following items before building the end-to-end automation path.

Requirement What to confirm
GitHub repository Repository exists at https://github.com/sautalwar/brandsafway-copilot-demo and has Copilot coding agent enabled.
Azure DevOps project Project is available at https://dev.azure.com/brandsafway1/Project1.
ADO ↔ GitHub connection Configured in Project Settings → GitHub connections so Azure DevOps can show Development links back to GitHub.
Repository secret Create ADO_PAT if you need bi-directional sync or follow-up updates back into Azure DevOps.
Labels Pre-create copilot-agent, ado-synced, severity:low, severity:medium, severity:high, and severity:critical.

Recommended token setup

For the Azure DevOps service hook call, create a GitHub PAT with permission to trigger repository dispatch events and interact with issues. Store it securely outside the HTML guide. Do not commit tokens or credentials into the repository.

3. Step-by-Step: ADO Service Hook Setup

Use an Azure DevOps service hook to send work item events directly to GitHub so the issue sync workflow runs automatically.

Create the subscription in Azure DevOps

  1. Open https://dev.azure.com/brandsafway1/Project1.
  2. Go to Project Settings → Service Hooks.
  3. Select Create Subscription.
  4. Choose Web Hooks as the service.
  5. Select the trigger Work item created or Work item updated, depending on when you want sync to occur.
  6. Apply filters such as work item type, area path, iteration path, or changed fields to narrow which items are sent to GitHub.

Configure the webhook target

  • Action URL: https://api.github.com/repos/sautalwar/brandsafway-copilot-demo/dispatches
  • Headers: Authorization: Bearer <GITHUB_PAT>
  • Headers: Accept: application/vnd.github+json
  • HTTP method: POST
  • Content type: application/json

Use this request body template

Map Azure DevOps work item fields into the GitHub repository_dispatch payload. Adjust the description or type mapping to fit your process.

{
  "event_type": "ado-work-item-created",
  "client_payload": {
    "ado_work_item_id": "{{workitem.id}}",
    "work_item_title": "{{workitem.title}}",
    "work_item_description": "...",
    "work_item_type": "User Story",
    "severity": "medium"
  }
}

Important implementation notes

  • Make sure the workflow is listening for the same event_type value, such as ado-work-item-created.
  • If you use Work item updated, avoid filters that create duplicate issue creation runs for non-actionable changes.
  • If severity is stored in a custom field, map it explicitly before the payload reaches GitHub labels.
  • Use the service hook test feature before enabling the subscription broadly.

4. Step-by-Step: Manual Trigger (workflow_dispatch)

If the ADO service hook is not ready yet, you can still prove the flow by manually running the GitHub Actions workflow.

Run the workflow manually

  1. Open the repository at https://github.com/sautalwar/brandsafway-copilot-demo.
  2. Select the Actions tab.
  3. Choose the workflow ADO Work Item to GitHub Issue Sync.
  4. Click Run workflow.
  5. Fill in the workflow inputs: work item ID, title, description, type, and severity.
  6. Click Run workflow again to submit the job.

Expected result

The workflow should create a GitHub issue that includes the Azure DevOps reference, apply ado-synced, copilot-agent, and the selected severity:* label, then assign the Copilot coding agent.

5. Verifying the Connection

After an automated or manual run, validate the setup from both systems.

Verification checklist

  • Check that a new issue was created in GitHub for the incoming Azure DevOps work item.
  • Verify the labels include ado-synced, copilot-agent, and the correct severity:low, severity:medium, severity:high, or severity:critical label.
  • Verify the Copilot coding agent is assigned to the issue.
  • Open the Azure DevOps work item and confirm the Development panel shows the linked GitHub issue.
  • Check the GitHub Actions run log for the ADO Work Item to GitHub Issue Sync workflow.

Traceability tip

Keep the AB#<work item id> reference in the created issue title or body so future branches, commits, and pull requests automatically link back into Azure Boards.

6. Troubleshooting

Most setup issues come down to permissions, payload formatting, or workflow expectations. Use the checks below to narrow the problem quickly.

Common issues

  • PAT permissions: The GitHub PAT cannot trigger the dispatch endpoint or create issues.
  • Webhook URL format: The dispatch endpoint path is incorrect or points to the wrong repository.
  • Label not found: The workflow expects labels that were not pre-created in the repository.
  • Copilot not available: Copilot coding agent is not enabled or cannot be assigned in the target repository.

How to inspect workflow logs

  1. Open Actions in GitHub.
  2. Select the latest run of ADO Work Item to GitHub Issue Sync.
  3. Open the failed job or step to inspect payload handling, label application, and assignment actions.

How to re-run a failed workflow

  1. Open the failed GitHub Actions run.
  2. Click Re-run jobs or Re-run failed jobs.
  3. If the failure came from bad payload data, correct the source work item or manual inputs first, then re-run.
  4. If the failure came from missing labels or permissions, fix the repository configuration before retrying.
