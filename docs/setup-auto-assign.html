1. Overview

A simpler, lower-friction path for getting from backlog item to working pull request.

What this path does

Manual Azure DevOps work item + manual GitHub issue + copilot-agent label → the auto-assign-copilot.yml workflow assigns GitHub Copilot to the issue automatically.

Why teams choose it

It is faster to set up, easier to explain in a live demo, and a strong fit for smaller teams that do not need full issue synchronization automation.

When to use this vs. the full automation path

Use this page when you want the simplest workable flow. Use setup-ado-sync.html when you want the fuller automation story with more orchestration between Azure DevOps and GitHub. For most demos, this manual-start / automated-assignment path is cleaner and more reliable.

2. Prerequisites

Complete these once before running the manual ADO + auto-assign workflow.

  • GitHub repository https://github.com/sautalwar/brandsafway-copilot-demo has the GitHub Copilot coding agent enabled.
  • Azure DevOps project https://dev.azure.com/brandsafway1/Project1 is connected to the GitHub repository so Azure Boards can detect AB# references.
  • The repository contains a label named copilot-agent.
  • The workflow file .github/workflows/auto-assign-copilot.yml exists in the repository.

Repository workflow

The workflow listens for GitHub issue events and checks for the copilot-agent label before assigning Copilot.

Important traceability note

For the Azure DevOps comment behavior in this repository, include AB#42 in the issue body. Including it in the title is still useful for visibility, but the workflow comment looks for AB# in the body.

3Step-by-Step: Create an ADO Work Item

2–3 min

Action

  1. Open Azure DevOps Boards: https://dev.azure.com/brandsafway1/Project1/_boards/board/t/.
  2. Create a new User Story, Bug, or Task.
  3. Enter the business request, expected outcome, and enough detail for an engineer or agent to act on it.
  4. Save the work item.
  5. Note the work item ID, for example 42.

Example

If Azure DevOps creates work item 42, that ID becomes the shared traceability token used in GitHub as AB#42.

Best practice

Keep the work item concise but outcome-focused. The GitHub issue will carry the richer implementation detail and acceptance criteria that the Copilot coding agent uses during execution.

4Step-by-Step: Create a GitHub Issue

3–4 min

Action

  1. Open the new issue page: https://github.com/sautalwar/brandsafway-copilot-demo/issues/new.
  2. In the title, include the Azure DevOps reference, for example: Add stats endpoint AB#42.
  3. In the body, include:
    • a short description of the requested change,
    • clear acceptance criteria,
    • the Azure DevOps reference again, such as AB#42.
  4. Add the label copilot-agent.
  5. Submit the issue.

Suggested issue body template

Summary
Add the requested enhancement for the inspection tracker API.

Acceptance Criteria
- Endpoint returns the required data
- Tests cover success cases
- Existing behavior remains unchanged

Traceability
AB#42

Critical detail

The repository workflow posts the Azure DevOps link comment only when it finds AB# in the issue body. Do not rely on the title alone for that part of the automation.

5. What Happens Automatically

After the issue is opened with the right label, the workflow takes over.

Automatic step What happens
Workflow trigger .github/workflows/auto-assign-copilot.yml runs on issues.opened when the issue already has the copilot-agent label, and also on issues.labeled if the label is added later.
Copilot assignment GitHub Copilot is assigned to the issue automatically.
Confirmation comment A comment is posted confirming that the Copilot coding agent has been assigned.
Azure DevOps note If an AB# reference is found in the issue body, the workflow adds a follow-up comment with the Azure DevOps work item link.
Agent execution The Copilot coding agent begins working: it reads the issue, creates a branch, implements the change, writes tests, and opens a pull request.

Workflow file location

For reviewers or demo participants who want to inspect the automation, point them to .github/workflows/auto-assign-copilot.yml in https://github.com/sautalwar/brandsafway-copilot-demo.

6. Verifying the Flow

Use these checkpoints to confirm the manual-start / auto-assign path is working correctly.

Check in GitHub

  • Open the issue and confirm Copilot is assigned.
  • Open the Actions tab and verify the Auto-assign Copilot agent workflow completed successfully.
  • Watch for the assignment comment and the Azure DevOps note comment.

Check in Azure DevOps

  • Open the work item in https://dev.azure.com/brandsafway1/Project1.
  • Check the Development panel for the linked GitHub issue.
  • Wait for later branch / PR artifacts to appear as Copilot works.

What to expect next

After assignment, allow time for the Copilot coding agent to create a branch, implement the requested change, add tests, and open a pull request in https://github.com/sautalwar/brandsafway-copilot-demo.

7. Label Management

The copilot-agent label is the switch that places an issue on the Copilot path.

Create the label in GitHub UI

Create copilot-agent from the repository label management area in GitHub (commonly Issues → Labels, or via repository settings access depending on your permissions).

Create the label with GitHub CLI

gh label create copilot-agent --repo sautalwar/brandsafway-copilot-demo --color 0f3d75 --description "Routes issues to the GitHub Copilot coding agent"

Add the label after issue creation

If the issue was created without copilot-agent, add the label later. That fires the labeled event, which the workflow also supports.

Other useful labels

  • ado-synced — shows the issue is part of the Azure DevOps traceability flow.
  • severity:low, severity:medium, severity:high — useful for demo filtering and prioritization.

8. Troubleshooting

Most issues come down to labels, event triggers, or the AB# reference format.

Copilot not assigned

  • Verify the label name is exactly copilot-agent.
  • Confirm the workflow has permission to edit issues.
  • Check the Actions tab for a failed or skipped run.

No Azure DevOps link comment

  • Confirm the issue body includes a valid reference such as AB#42.
  • Check for typos like AB 42, #42, or lowercase variations that do not match the expected format.

Workflow did not trigger

  • Open the Actions tab in https://github.com/sautalwar/brandsafway-copilot-demo.
  • Verify the issue event was opened with the label present, or labeled when the label was added later.
  • Confirm .github/workflows/auto-assign-copilot.yml exists on the default branch.

Fast demo recovery

If a live demo misfires, add or remove/re-add the copilot-agent label to trigger the labeled path again, then refresh the issue and Actions tab.
